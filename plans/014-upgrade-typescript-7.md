## Why this matters

The repo pins `typescript: ^6.0.3`; the current npm `latest` is `7.0.2` — the
native (Go) rewrite of the compiler, ~10x faster type-checking. This repo's
`tsconfig.json` is already aligned with TS 7's stricter defaults
(`moduleResolution: "bundler"`, `target: "ES2020"`, `esModuleInterop: true`,
`strict: true`, no `baseUrl`, no `downlevelIteration`, relative `paths`,
explicit `types: ["chrome"]`), so config-wise the upgrade should be small. The
real unknown is behavioral/tooling: TS 7 does not yet expose a stable
programmatic compiler API, so tools that embed the TS API (notably Storybook's
`react-docgen-typescript`, and typescript-eslint / Volar-style editors) may not
work under TS 7 yet. This plan bumps the version and **validates empirically via
the repo's full gate**, then reports exactly what passes and what breaks so the
reviewer can decide whether to adopt now or wait for the ecosystem.

## Steps

### Step 1: Branch and bump the version

```bash
git checkout -b advisor/014-upgrade-typescript-7
```

Edit `package.json`: change `"typescript": "^6.0.3"` to `"typescript": "^7.0.2"`.
Then:

```bash
pnpm install
pnpm exec tsc --version
```

**Verify**: `pnpm install` exits 0; `tsc --version` prints `Version 7.0.x`. If pnpm
cannot resolve `typescript@^7.0.2` (registry/policy — note the repo enforces a
`minimum_release_age` for deps, so a very new 7.0.x may be blocked), STOP and
report the resolution error (do not disable the trust/age policy).

### Step 2: Typecheck under TS7

```bash
pnpm run typecheck
```

**Verify**: exit 0, no errors. If NEW type errors appear that did not exist on
6.0.3, capture them verbatim and STOP — report whether they look like (a) a
config rejection (fixable minimally in tsconfig, in scope) or (b) genuine source
type errors (out of scope to fix; report the file:line list).

### Step 3: Lint, unit tests, build

```bash
pnpm run lint
pnpm run test
pnpm run build
```

**Verify**: each exits 0 / all tests pass. `dist/content.js` should still build
(and stay minified — the esbuild config is independent of the TS version).

### Step 4: Storybook (the highest-risk stage)

```bash
pnpm run test:storybook
```

**Verify**: all Storybook tests pass. If this stage fails with an error
originating in `react-docgen-typescript` / a "programmatic API" / TS API version
mismatch (e.g. Storybook cannot load the TS 7 compiler), STOP and report the
exact error. This is the expected failure mode if the ecosystem is not ready —
it is a REPORTABLE outcome, not a defect to fix here.

### Step 5: Full gate

```bash
mise run ci
```

**Verify**: all stages pass end-to-end. Only if this is fully green do you commit.
